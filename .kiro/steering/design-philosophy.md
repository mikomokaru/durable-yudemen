# 設計哲学 — 真善美は一つの判断である

この文書は、本コードベースにおけるすべての設計・実装・レビュー判断の最終的な拠り所である。
Kiro はこれを内面化し、コード・コメント・コミット・spec ドキュメント（design / tasks）の全てにおいて一貫して従う。

## 序：真善美は三つの目標ではなく、一つの判断である

コードにおいて、真・善・美は別々のチェック項目ではない。一つの判断の三つの側面である。

- 正しいコードは単純へ向かう。
- 単純なコードは美しさへ向かう。
- 美しいコードは、自らの正しさを露わにする。

だから美を直接狙わない。狙えば逃げる。clever さ、過剰な抽象、自己満足の層が忍び込む。
代わりに、真と善を妥協なく追求する。美は、その副産物として現れる。

この三つが衝突して見えるとき、それはどれか一つを誤解している。
嘘（真の欠如）は必ず醜く（美の欠如）、無駄（善の欠如）は必ず醜い。
三つのテストは、一つの判断の三つの視点にすぎない。

各原則は哲学で終わらない。直後に、各行で従える規則へ換金される。

## 真 — コードは真実を語る

コードは、現実を正確に写し、自らの状態について決して嘘をつかない。

- **メモリと永続層の区別を偽らない。** Working_Copy への代入は永続化ではない。メモリに入れたものを「保存された」かのように扱わない。確定の起点は `storage.put` の成功のみ。この区別を曖昧にするコードは、状態について嘘をついている。
- **永続層が唯一の正本（SSOT）。** broadcast も発火も、永続が確定した事実の上にのみ立つ。`put` 成功の前に外部へ真実を主張しない。
- **不正な状態を表現可能にしない。** 「あり得ない状態」を型で排除する。endTime を持たない Timer、Slot に属さない Timer——それらが構築できてしまう設計を許さない。バリデーションで弾くより、構築不能にする方が真である。
- **全てのパスを構造で表現する。** 拒否・失敗・上限超過——requirements が挙げた全ての分岐は、「後で考える例外」ではなく、戻り値と型に最初から織り込む。部分関数を避け、握り潰された失敗を残さない。
- **隠れた状態を持たない。** 可変な隠し状態は、読み手を欺き、かつ絡まる。状態は明示的で、検査可能で、一望できる。
- **時間の真実。** endTime は絶対時刻という「事実」。残り秒は、その事実からの「導出値」であって状態ではない。状態として残り秒を持てば、二つの真実の源が生まれ、必ずズレる。

## 善 — コードは善く仕える

コードは、その目的に善く仕え、誰も害さない——使い手を、運用者を、未来の保守者を、システムを。

- **厨房スタッフへの善。** 失敗は優雅に劣化する。瞬断で表示は死なない（endTime のローカル計算）。失敗を握り潰さず、回復経路を持つ（broadcast 失敗は再接続 hydration が回収する）。現場の速度を奪わない。
- **運用者への善。** hibernation の規律は、資源を浪費しないという倫理である。「待つなら寝かせる仕組み（Alarm / WS Hibernation / Queue）に逃がす。抱える（setInterval / waitUntil / 外部 await）と漏れる。」——この一本の原則が、コストと善を同時に守る。
- **未来の保守者への善。** コードは自己説明的であれ。コメントは「何を」を語らない——それは名前と構造で語る。コメントが許されるのは「なぜ」だけ。自明でない設計判断の理由（例：「Alarm は早期発火しうるので許容窓を持つ」）。コメントを書きたくなったら、まず名前と分割で消せないか試す。
- **システムへの善＝構造の主権。** ロジックとプラットフォームを分離する（PoA / PoR）。DO 固有の調整機構への依存を、業務ロジックから隔離する。ロジックは純粋関数として、いつでも他基盤へ運べる形に保つ。ロックインは「配線」レベルに封じ込め、事業の核には及ばせない。差し替え可能であること——レゴ式再構成性——が善である。
- **善は一貫性。** 全コードは少数の原則から演繹される。各メソッドが、どの原則のどの帰結かを、構造自体で示す。継ぎ接ぎではなく、一つの思想の展開であれ。

## 美 — コードは必然である

美は足されない。それ以上何も取り除けなくなったとき、残るものが美である。

- **inevitable であって clever でない。** 目指すのは「これ以外あり得ない」という感覚。技巧の誇示ではなく、問題の本質への最短の構造。読んだ人間が「自分でもこう書いたはずだ」と感じるなら、それは美しい。
- **抽象は、重複が実在してから入れる。** 将来のための層を先回りで作らない（YAGNI）。一度しか現れていない概念に、再利用のための抽象を被せない。
- **パターンを名前で持ち込まない。** 「ここは Strategy パターンで」とは考えない。問題が自然にその形を要求したとき、結果としてその形になる。形が先にあってはならない。
- **計算と作用を分離する。** 純粋な計算（最早 endTime の算出、状態遷移の決定）と、副作用（`storage.put`、broadcast、`setAlarm`）を混ぜない。計算は純粋に、作用は端へ寄せる。「角度を変える手続き」が要らない構造——変換の儀式が現れたら、境界の引き方を疑う。
- **重複の根絶。** 同じ概念は、ただ一箇所で定義する。endTime の扱い、最早 Timer の算出、broadcast の形——コードベース全体で一度だけ。二度書かれた概念は、二つの真実になりかけている。
- **美は狙うと逃げる。** だから美のための行を一行も書かない。真と善を徹底し、その残響として美が立ち現れるのを待つ。

## 三位一体 — なぜ三つで一つか

`setInterval` でカウントダウンを回す実装を考えよ。

- それは hibernation を妨げて資源を浪費する（**善の欠如**）。
- それは「サーバが時を刻む」という余計な状態を持ち、残り秒という導出値を状態に昇格させて二つの真実を生む（**真の欠如**）。
- そしてそれは、本来クライアントのローカル計算で消えるはずの複雑性を抱え込み、絡まる（**美の欠如**）。

一つの誤りが、三つの欠如として同時に現れる。

逆もまた然り——endTime を絶対時刻で持ち、秒読みをクライアントに委ね、発火を Alarm に逃がす一つの正しい判断は、真であり、善であり、美である。

三つのテストは、一つの判断の三つの視点である。
一つに適って二つに反するように見えたら、立ち止まれ。その判断を誤解している。

## 最終審級 — 誰が美を所有するか

この原則に忠実なコードを、Kiro は生成できる。正直で、善く仕え、削ぎ落とされた素材を。
だが、マスターピースの最後の数パーセント——「これ以外あり得ない」の彫り込み——は、人間の構造感覚にしか宿らない。言語化しきれない残余が、必ずある。美の最終審級は、書き手の内部にある。

ゆえに Kiro の役割は「美しいコードを完成させる」ことではない。人間が彫れる、素直で原則に忠実な素材を差し出すことである。
最適化すべきは「印象的であること」ではなく「彫りやすいこと」——正直で、単純で、原則に忠実で、人間が後から取り消すべき抽象が少ないこと。

マスターピースは、生成ではなく、レビューと書き直しの往復から生まれる。

## レビューの作法 — 引き算であって足し算ではない

レビューは何も足さない。原則違反を取り除くだけである。
問いは決して「何を足せば良くなるか」ではない。「何を取り除けば、必然だけが残るか」である。

- **真の検査：** これは状態について嘘をついていないか。隠れた可変状態はないか。握り潰された失敗パスはないか。導出値が状態に昇格していないか。
- **善の検査：** これは hibernation を妨げていないか。劣化を優雅に保っているか。このコメントは、名前か構造で消せないか。ロジックがプラットフォームに癒着していないか。
- **美の検査：** これは取り除けるか。この抽象は、その存在の代償に見合う実在の重複に支えられているか。この概念は二度定義されていないか。

取り除けるものが何もなくなったとき、コードは真であり、善であり、美である。そこで初めて、完成と呼ぶ。

## 流れ — 導出と変換、しかし機械は持ち込まない

このコードベースは、既に「流れ」として考えられている。状態が流れ、そこから値が導かれ、イベントが状態を変える。これは FRP（関数型リアクティブ）の核である。だが核（思想）と機械（ストリームライブラリ）は別物だ。思想を中核に据え、機械は持ち込まない。

この区別を明文化するのは、ここが最も脱線の引力が強いからである——「リアクティブにしたい」という正しい直感が、不要な重機を呼び寄せる。中核は二つの規律に尽きる。

### 導出値は状態の関数である

ある値が他の値から計算できるなら、それは状態ではない。残り秒は状態ではない。endTime（事実）と現在時刻（時間）からの導出値 `remaining = endTime - now` である。

導出値を状態に昇格させない。状態として保持した瞬間、二つの真実の源が生まれ、それを揃えるための同期が要る（真の欠如）。同期のために `setInterval` が呼び込まれる（善の欠如）。そして絡まる（美の欠如）。一つの昇格が、三つの欠如を呼ぶ。

- **規則：** 計算できるものは、保持せず計算せよ。状態は「これ以上分解できない事実」だけに絞る。endTime は状態、残り秒は関数。

### 状態遷移は純粋な変換である

状態を変えるのは、ただ一つの形——`(現在の状態, イベント) → 新しい状態` という純粋関数。タイマー開始も、キャンセルも、発火処理も、全てこの形の変換に還元する。

変換は純粋であれ。同じ入力に同じ出力。変換の内部で `put` も broadcast も `setAlarm` もしない。代わりに、変換は「次に実行すべき作用」をデータとして記述して返す。端（shell）が、返された作用の記述を実行する。

状態を決める純粋な計算と、世界を変える副作用を、同じ関数に混ぜない。これが計算と作用の分離であり、「角度を変える手続きがない」構造の正体である。変換は何が起きるべきかを語り、shell がそれを起こす。

- **規則：** 状態を決める関数と、世界を変える関数を、決して同じ関数の中で混ぜない。

### しかしストリームライブラリは core に持ち込まない

上の二つは FRP の核そのものである。だが RxJS 等のストリーム機械を StoreTimerDO の中核に入れてはならない。これは三位一体のテストに、三つとも落ちる。

- **美の欠如：** DO が捌くのは数種の離散イベントを単一スレッドで順に処理するだけ。Observable / Subject / operator / Subscription の機構は、存在しない複雑さへの抽象である（YAGNI）。多ストリーム合成という、ここに無い問題のための道具。
- **真の欠如：** Subscription は見えにくい生きた状態を抱える。状態を一望できる透明性が濁る。しかも Observable の購読は in-memory の生きた構造であり、hibernate / rehydrate を跨げない。
- **善の欠如：** 時間系オペレータ（`interval` / `timer` / `debounceTime`）は内部で `setInterval` を回し、hibernation を殺す。最も使いたい部分が、最も使えない。

一つの機械が、三つの欠如を同時に呼ぶ。三位一体の、ちょうど裏返し。

ゆえに——思想を採り、機械を捨てる。ライブラリへ手が伸びる衝動は、思想が正しい証拠であって、機械が要る証拠ではない。導出値と純粋変換は、ライブラリなしの、ただの関数として書ける。それで足り、それが美しい。

（もしストリームライブラリが正当化される日が来るなら、それは DO ではなく React 側の端である。クライアントは WS・タップ・アニメーションフレームという本物の多ストリームを持つ。だが core はここに、思想だけを置く。）

## 中核語彙（このコードベースの母語）

設計判断は、以下の語彙で記述・参照される。Kiro はこれらを内面化し、コード・コメント・コミットにおいて一貫して用いる。

- **構造の主権（Structural Sovereignty）**：ベンダーロックインを境界に封じ込め、核を自由に保つ。
- **PoA / PoR 分離**：処理の権限（Authority）と記録の権限（Record）を意識的に配置する。
- **レゴ式再構成性**：各単位は同型で交換可能。継ぎ目で分解・再構成できる。
- **SSOT**：真実の源は一つ。永続層が状態の正本、組織マスタが階層の正本。
- **「待つなら寝かせる、抱えると漏れる」**：hibernation 規律の一本道。
- **「角度を変える手続きがない」**：変換の儀式が現れたら、境界の引き方を疑え。
